## 使用說明

### 如何準備這些題目

1. **STAR 原則**：行為面試題建議用 STAR 原則回答

   - **S**ituation（情境）：描述當時的背景
   - **T**ask（任務）：說明你面對的任務或挑戰
   - **A**ction（行動）：詳述你採取的具體行動
   - **R**esult（結果）：說明最終成果和學到的教訓

2. **具體化回答**：用數據和實際案例支持你的論點

   - ❌ 不好：「我很注重程式碼品質」
   - ✅ 好：「在 VoiLoad 專案中，我透過建立 E2E 測試覆蓋率從 0% 提升到 85%，減少了 60% 的線上 bug」

3. **展現思維過程**：工程能力問題不只要說「是什麼」，更要說「為什麼」和「何時使用」

4. **誠實面對不足**：不知道的就說不知道，但要展現學習意願和如何快速學習

5. **水球文化關鍵字**：準備時留意以下水球軟體學院重視的特質
   - **主動性**：主動釐清需求、主動溝通（不主動提問會直接落選）
   - **First Principles Thinking**：從基本原理重新思考問題
   - **Validate & Iterate**：Test-Learn 思維，快速驗證與迭代
   - **透明度**：公開進度、Checkpoint 機制、頻繁同步
   - **Storytelling**：清楚陳述問題、假設、行動、驗證

**First Principles 範例**：

```
問題：Chrome 擴充功能無法下載 Facebook 語音訊息
舊思維：找現成的下載套件
First Principles：
  1. 基本原理：語音訊息本質是音訊檔案的 URL
  2. 拆解假設：為什麼抓不到 URL？（被 FB 加密？動態產生？）

結果：VoiLoad 擴充功能，1000+ 用戶
```

### Code Quality 與測試策略

#### Q15. 你認為什麼時候該寫測試？單元測試、整合測試、E2E 測試如何取捨？

**評估重點**：

- Test-Learn 思維
- 測試策略的務實性
- 不過度設計

**準備方向**：

- 測試金字塔（Unit > Integration > E2E）
- 何時該寫測試（核心邏輯、容易出錯的地方、Public API）
- 何時可以不寫測試（快速驗證的 POC、一次性腳本）
- Requirement 提到「一定要有 E2E」的原因

**參考答案**：

```
優先順序：
  1. E2E 測試：確保核心用戶流程可用（註冊、登入、購買）
  2. 單元測試：複雜的業務邏輯（計算、驗證、轉換）
  3. 整合測試：API、資料庫互動
快速迭代時：
  - Release 1：只寫關鍵 E2E，確保基本可用
  - Release 2：補單元測試，提升信心
  - Release 3：整合測試，覆蓋邊界情況
```

#### Q16. 在 Requirement 中提到「請勿做 clean-arch 等進階架構設計」，你如何理解這句話？

**評估重點**：

- KISS 原則的理解
- 不過度設計
- 務實的架構思維

**準備方向**：

- Clean Architecture 的優缺點
- 什麼情況適合複雜架構？什麼情況不適合？
- 「基本分層」vs「過度設計」的界線
- 如何在簡單和可擴展之間平衡

**參考答案**：

```
理解：
  1. 時間限制（一週半），沒時間做複雜架構
  2. 面試官比你熟架構，過度設計是危險地帶
  3. MVP 思維：先求可用，再求優雅
  4. 基本分層即可：Controller - Service - Repository

務實做法：
  - 有基本的關注點分離（前後端分離、資料存取層）
  - 不要為了「展現能力」而過度抽象
  - 保持程式碼可讀性和可測試性即可
```

### AI x BDD 與規格驅動開發

#### Q25. 你對「規格驅動開發」（BDD）有什麼理解？是否有實踐經驗？

**評估重點**：

- BDD 認知（水球文化核心）
- 可執行規格

**準備方向**：

- BDD vs TDD 的差異
- Gherkin 語法（Given-When-Then）
- 如何寫可執行規格

**參考答案**：

```
BDD 理解：
```

## 面試技巧提醒

### 展現水球文化契合度

1. **主動性**

   - ✅ 強調「主動釐清需求」「主動提問」「主動同步」
   - ✅ 分享你如何「無數次溝通」來確保理解正確
   - ❌ 避免給人「被動等待指示」的印象

2. **First Principles Thinking**

   - ✅ 展現你如何「從基本原理重新思考問題」
   - ✅ 用具體案例說明你如何 Break down → Rebuild
   - ❌ 避免只套用既有解法，缺乏深度思考

3. **Test-Learn-Iterate**

   - ✅ 強調「快速驗證、從失敗中學習、持續迭代」
   - ✅ 分享失敗經驗和復盤過程
   - ❌ 避免只談成功案例，忽略失敗學習

4. **透明度與 Checkpoint**

   - ✅ 展現你如何「公開進度、頻繁同步、及時回報」
   - ✅ 使用工具（Kanban、Dashboard、Daily note）的經驗
   - ❌ 避免給人「黑箱作業」的印象

5. **Storytelling**
   - ✅ 清晰陳述「問題 → 假設 → 行動 → 驗證」
   - ✅ 用數據和具體案例支持論點
   - ❌ 避免模糊、抽象、缺乏細節的回答

### 回答技巧

1. **具體化**：用數字、案例、截圖讓答案更有說服力
2. **誠實**：不知道就說不知道，但展現學習意願
3. **簡潔**：先給結論，再補充細節（面試官會追問）
4. **連結經驗**：盡量連結到你的實際專案（VoiLoad、HPB、線上教學）
5. **展現熱情**：對技術、學習、成長的熱情

### 準備檢查清單

- [ ] 每題都準備了具體案例（不只理論）
- [ ] 準備了 STAR 格式的回答（Situation, Task, Action, Result）
- [ ] 準備了數據支持（用戶數、效能提升、時間節省）
- [ ] 練習了 Storytelling（清晰陳述問題和解決方案）
- [ ] 準備了「你有什麼問題要問我們」（展現主動性）
